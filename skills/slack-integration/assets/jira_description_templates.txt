# Jira Description Templates for Slack-Sourced Issues

These templates follow Jira best practices: ELI5 summary, concise acceptance criteria, and links to detailed context.

*Reported By*: @<slack_username>
*Date*: <date>

h3. Issue Description
[Concise description of the problem]

h3. Steps to Reproduce

# Step 2
# Step 3

h3. Expected Behavior
[What should happen]

## Usage Guidelines

### Choosing a Template

1. **Bug Report** - Use when Slack thread describes a defect or error
2. **Feature Request** - Use when Slack thread proposes new functionality
3. **Technical Debt** - Use when Slack thread discusses code improvements or refactoring
4. **Investigation/Spike** - Use when Slack thread raises questions requiring research
5. **Incident** - Use when Slack thread is about a production issue or outage
6. **Decision** - Use when Slack thread reached a decision that needs tracking
7. **Customer Feedback** - Use when Slack thread contains customer input
8. **Simple Task** - Use for straightforward tasks that don't fit other categories

### Filling Out Templates

**Summary (ELI5)**:
- Start with the most important information
- Explain in simple terms (Explain Like I'm 5)
- 1-2 sentences maximum
- Anyone should understand without technical background

**Context from Slack**:
- Always include the Slack thread link
- List key participants (@ mentions)
- Include date for time context
- Add any relevant metadata (severity, priority, etc.)

**Discussion Summary**:
- Extract key points from the thread
- Focus on conclusions, not play-by-play
- Highlight important decisions or insights
- Keep it concise - full details are in Slack link

**Acceptance Criteria**:
- Make each criterion specific and testable
- Use numbered list for clarity
- Keep each point brief (one line if possible)
- Focus on "what" not "how"

**Additional Context**:
- Always refer back to Slack for full details
- Mention if there are screenshots, logs, or diagrams in Slack
- Don't duplicate everything from Slack - link to it

### Common Mistakes to Avoid

**DON'T**:
- Copy entire Slack conversation into Jira
- Write verbose paragraphs
- Include off-topic discussion
- Forget to link back to Slack
- Make assumptions without checking Slack thread
- Leave placeholders like <slack_thread_url> unfilled

**DO**:
- Summarize key points concisely
- Link to Slack for full context
- Extract actionable acceptance criteria
- Keep it scannable and easy to read
- Update template with actual values
- Follow Wiki syntax (not Markdown)

### Wiki Syntax Reminders

When filling templates, use correct Wiki syntax:

- Headers: `h3. Header` (not `### Header`)
- Bold: `*bold*` (not `**bold**`)
- Links: `[text|url]` (not `[text](url)`)
- Code: `{{code}}` (not `` `code` ``)
- Lists: `* item` or `# item` (same as Markdown)

### Automation Tips

When using these templates programmatically:

1. Parse Slack URL first
2. Fetch thread content with `slack_get_thread_replies`
3. Analyze content to determine appropriate template
4. Extract key information (participants, date, summary)
5. Fill in template with extracted data
6. Save to file for review
7. Create Jira issue with `jira_create.py`
8. Reply to Slack thread with Jira link

Example automation:
```bash
# 1. Parse URL
python3 parse_slack_url.py "$SLACK_URL" > /tmp/parsed.json

# 2. Extract data
CHANNEL_ID=$(jq -r '.channel_id' /tmp/parsed.json)
THREAD_TS=$(jq -r '.thread_ts' /tmp/parsed.json)

# 3. Fetch thread (via MCP tool - use Claude to call)
# slack_get_thread_replies({"channel_id": "$CHANNEL_ID", "thread_ts": "$THREAD_TS"})

# 4. Analyze and fill template (AI-assisted)

# 5. Create Jira issue
python3 ../jira-integration/scripts/jira_create.py \
  --project GS \
  --summary "Title from Slack" \
  --description-file /tmp/jira_desc.txt \
  --type Story

# 6. Reply to thread
# slack_reply_to_thread({"channel_id": "$CHANNEL_ID", "thread_ts": "$THREAD_TS", "text": "Created GS-12345"})
```
